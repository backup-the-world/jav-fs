# PRD：提升 JAV 视频重复扫描准确性

## Problem Statement

用户希望通过重复扫描模式检查 incoming 媒体目录中是否存在重复 JAV 视频文件。但当前扫描结果曾经不可用：程序会把下载站前缀误识别为番号 ID。

例如：

- `hhd800.com@MIDA-307.mp4` 被错误识别为 `hhd800`
- 大量来自同一下载站的不同视频被归为同一个 ID
- `actual videos` 数量严重偏低
- `duplicate videos size` 被严重夸大
- `Conflicts` 列表包含大量误报，无法作为清理依据

修复下载站前缀问题后，又出现另一类误报：多分片视频。例如：

- `13dsvr01794_1_8k.mp4`
- `13dsvr01794_2_8k.mp4`

这类文件是同一作品的不同分片，不应被视为重复文件。重复扫描必须避免引导用户误删有效分片。

## Solution

改进重复扫描逻辑，让程序明确区分三个概念：

1. 从带噪声的下载文件名中提取标准视频 ID。
2. 按规范化后的视频 ID 分组检测重复。
3. 识别多分片视频，避免把同一作品的不同分片报告为重复。

从用户视角看，`--show-duplicate` 应只报告“很可能可以删除的重复副本”，而不是：

- 下载站前缀导致的错误重复；
- 同一作品的不同有效分片。

扫描仍应保持当前 CLI 使用方式、并发扫描性能和现有 summary 输出结构。

## User Stories

1. 作为媒体库维护者，我希望程序能从 `hhd800.com@MIDA-307.mp4` 中提取 `MIDA-307`，以免下载站前缀造成误报。
2. 作为媒体库维护者，我希望程序能从 `4k2.me@mida-307.mp4` 中提取 `mida-307`，以便不同下载站来源的同一作品能正确归组。
3. 作为媒体库维护者，我希望重复扫描对番号 ID 做大小写归一化，以便 `mida-307` 和 `MIDA-307` 被视为同一作品。
4. 作为媒体库维护者，我希望程序优先识别带横杠的番号 ID，以免先匹配到 `hhd800` 这类站点 token。
5. 作为媒体库维护者，我希望程序支持 `T38-052` 这类字母数字混合前缀，以免合法番号被错误解析。
6. 作为媒体库维护者，我希望程序支持数字站点名前缀后的番号，例如 `4k2.com@13dsvr01794_2_8k.mp4`，以便正确提取作品 ID。
7. 作为媒体库维护者，我希望 `_1_8k` 和 `_2_8k` 被识别为分片标记，以免不同分片被报告为重复。
8. 作为媒体库维护者，我希望 `_1` 和 `_2` 被识别为分片标记，以支持没有清晰分辨率后缀的多分片文件。
9. 作为媒体库维护者，我希望 `CD1` 和 `CD2` 被识别为分片标记，以支持旧式多碟命名。
10. 作为媒体库维护者，我希望同一番号下、分片号不同的两个文件不被报告为重复，以免误删有效分片。
11. 作为媒体库维护者，我希望同一番号下、分片号相同的两个文件仍被报告为重复，以便找出同一分片的重复副本。
12. 作为媒体库维护者，我希望普通文件名和下载站前缀文件名的同一番号仍被报告为重复，例如 `JUR-505.mp4` 与 `hhd800.com@JUR-505.mp4`。
13. 作为媒体库维护者，我希望无法提取番号的视频被计入 failed videos，以便后续发现未覆盖的命名模式。
14. 作为媒体库维护者，我希望 duplicate size 只统计可能重复的副本，以便估算可释放空间。
15. 作为媒体库维护者，我希望 Conflicts 列表只包含可行动的重复候选，以便人工清理。
16. 作为 CLI 用户，我希望命令行参数保持不变，以便现有使用习惯不受影响。
17. 作为 CLI 用户，我希望扫描挂载卷时仍保持较快速度，以便大目录扫描仍然实用。
18. 作为开发者，我希望文件名解析逻辑有单元测试覆盖，以免后续改动重新引入误报。
19. 作为开发者，我希望多分片判断封装成可独立测试的接口，以便不依赖文件系统遍历就能验证行为。
20. 作为开发者，我希望重复归类决策足够明确，以便区分“ID 提取错误”和“重复判断错误”。

## Implementation Decisions

- 构建或完善一个较深的文件名解析模块，提供稳定、可测试的接口：
  - 从文件名提取标准视频 ID；
  - 从 ID 提取 prefix；
  - 从文件名提取可选分片号；
  - 判断同一 ID 下的两个文件是重复副本还是不同分片。
- 重复扫描在写入重复检测 map 前，应将视频 ID 统一转成大写。
- ID 提取应先去除视频扩展名，再在存在 `@` 时优先解析最后一个 `@` 后面的部分。
- ID 提取应优先识别带横杠 ID，再识别不带横杠 ID，避免 `hhd800` 先于 `MIDA-307` 被匹配。
- 带横杠 ID 的 prefix 应允许字母数字混合，但必须至少包含一个字母，以支持 `T38-052`。
- 不带横杠 ID 应继续支持 `XYZ456`、`dsvr01794` 等既有模式。
- 同一标准 ID 下，如果两个文件都有显式分片号，且分片号不同，则不视为重复。
- `START-476`、`T38-052` 这类普通番号末尾数字不应被误判为分片号。
- 如果没有明确证据证明两个同 ID 文件是不同分片，则仍按重复候选处理。
- 并发文件系统遍历和进度条模型保持不变。
- 现有输出结构保持不变：先输出 summary，再输出 conflict list。

## Testing Decisions

- 测试应验证外部行为，不绑定具体正则实现细节。
- 文件名解析应覆盖真实下载文件名：
  - `hhd800.com@MIDA-307.mp4`
  - `4k2.me@mida-387.mp4`
  - `4k2.com@13dsvr01794_2_8k.mp4`
  - `T38-052.mp4`
  - `ABC-123.mp4`
  - `XYZ456.mp4`
  - `XYZ-123-CD1.mp4`
- 多分片识别应覆盖：
  - `_1_8k`
  - `_2_8k`
  - `CD1`
  - `CD2`
- 多分片识别必须明确测试普通番号末尾数字不会被当成分片号。
- 重复分类应覆盖：
  - 不同显式分片号 → 不重复；
  - 相同显式分片号 → 重复；
  - 普通文件 + 下载站前缀文件 → 重复。
- 保留现有 SMB URL 转换、视频文件识别、图片文件识别、ID 提取、prefix 提取等单元测试。
- 可增加一个 CLI 级回归检查，使用真实命名模式 fixture，确认已知多分片文件不会出现在 conflicts 中。

## Out of Scope

- 自动删除重复文件。
- 自动移动重复文件到隔离目录。
- 交互式重复文件处理器。
- 从外部 JAV 数据库获取元数据。
- 覆盖所有可能的 JAV 命名约定；本 PRD 只覆盖已观察到并确认的问题模式。
- 修改 SMB 认证行为。
- 修改扫描并发架构。
- 修改 prefix scan 语义，除非它自然复用改进后的 ID 提取逻辑。
- 将扫描结果持久化到数据库或缓存。

## Further Notes

本次问题的影响很明显：修复前，在 1653 个视频文件中，重复扫描只识别出 146 个 actual videos，并估算出 8.20 TB duplicate videos size。修复下载站前缀和多分片识别后，actual videos 上升到 1634，duplicate size 降到约 92.76 GB，剩余 conflicts 看起来更接近真实重复副本。

后续可以考虑把重复判断进一步抽取为独立的 duplicate classification 模块。该功能的核心价值不在文件系统遍历，而在稳定、可测试地判断两个文件名代表：

- 同一个副本；
- 不同副本；
- 同一作品的不同有效分片。
